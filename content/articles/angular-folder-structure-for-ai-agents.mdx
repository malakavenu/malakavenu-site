Across every Angular project I've shipped over the last few years — startup MVPs, enterprise migrations, internal tools, side projects — I quietly converged on the same `src/app/` layout. Not because I read it in a book. Because I kept hitting the same wall.

The wall is always the same. As soon as a project gets serious — multiple features, multiple engineers, real customers — the flat `components/services/models/` layout most Angular tutorials still teach starts costing real time. Files pile up in three folders that have no relationship to the business. Naming gets defensive (`OrderListComponent`, `OrderDetailComponent`, `OrderCheckoutComponent`, `OrderConfirmationComponent`...). New engineers ask "where does X live?" forty times a week.

Eventually I gave up fighting it and committed to a structure I now use everywhere:

```text
src/app/
├── core/         ← app-wide singletons (services, interceptors, guards, models)
├── shared/       ← reusable, stateless UI primitives (components, directives, pipes)
├── features/     ← one folder per business capability — auth, dashboard, products
├── layouts/      ← app shells and templates (main-layout, auth-layout)
├── state/        ← global state only; feature state stays inside features/
└── app.routes.ts
```

Each `features/` folder owns its own components, services, routes, and (with Angular 21 signals) its own state. `core/` is for things there should only be one of. `shared/` is for things that have no domain meaning but get reused. `layouts/` for shells. `state/` for genuinely global state.

For a long time I treated this purely as _"the way to scale Angular for a team."_ Then last year — working with Cursor and Claude Code daily across these same repos — I noticed something I hadn't expected.

> The same folder structure that scales for a team of fifty also scales for a fleet of AI agents. The flat layout teaches the model bad habits. This layout teaches it the conventions I would have spent forty-five minutes explaining in onboarding.

A few weeks ago I wrote about [the six files that turn a codebase AI-ready](/articles/ai-ready-codebase-structure) — the artifacts that sit _next to_ your code. This post is about the layer almost nobody talks about: **the directory tree itself**.

## What the model sees when it opens your repo

Drop Claude Code, Cursor, or Copilot into a fresh Angular project and watch what happens in the first thirty seconds. It indexes the workspace. It builds a mental map of where things live. Every suggestion it makes in this repo — for the next year — is filtered through that first impression.

I've watched this play out side-by-side in two of my own projects, one with the flat layout and one with the `core/shared/features/` layout. The difference is not subtle.

**Flat Angular (`src/app/components/`, `src/app/services/`, `src/app/models/`)**

The model sees a repo with **three responsibility silos** and zero domain language. When I ask it to "add a checkout flow," it has nowhere to put it. So it does what beginners do — drops a `CheckoutComponent` in `components/`, a `CheckoutService` in `services/`, and a `CheckoutModel` in `models/`. By the time the project has ten features, the `services/` folder has 47 files and the model is guessing at imports because every file has a generic name and no neighborhood.

This is the AI equivalent of a junk drawer. The model technically _can_ find things. It just can't _reason_ about them.

**Feature-folder Angular (the structure above)**

Now the model sees **a domain map**. When I ask it to "add a checkout flow," there's exactly one correct answer: a new sibling under `features/`, owning its own components, services, routes, and state. When I ask "where does auth live?" — `features/auth/`. When I ask "is this a singleton?" — if it's in `core/`, yes; if not, no.

Every question has a structural answer. The model isn't guessing. It's reading a contract.

That contract is **code context**, encoded in the most durable medium a codebase has: file paths.

## Why this matters more in 2026 than it did in 2024

Three things changed in the last eighteen months that turned folder structure from a style preference into load-bearing infrastructure for me.

**1. Agents now act on multiple files at once.** A single Cursor or Claude Code agent run regularly touches 8 to 30 files in my projects. If those files are scattered across flat `components/` and `services/` folders, the agent has to load and re-load context constantly. If they're co-located inside `features/checkout/`, one folder read covers the whole task. Folder boundaries are **token boundaries**.

**2. Standalone components and signals removed the safety net.** NgModules used to give Angular projects a free organizational layer. In [Angular 21 — standalone by default, zoneless by default, signals everywhere](/articles/angular-21-signals-zoneless) — that layer is gone. The folder structure now _is_ the architecture. If I don't draw a feature boundary explicitly with a folder, nothing else draws it for me.

**3. Module-boundary lint rules made the contract enforceable.** Tools like Nx's `enforce-module-boundaries`, ESLint's `import/no-restricted-paths`, or `dependency-cruiser` let me say _"nothing in `features/billing/` can import from `features/auth/`."_ Once the rule exists, the agent reads it, the agent respects it, and the agent — not just the human — keeps the boundaries intact during refactors. Folders become not just organizational, but _executable_.

## The structure, re-read as a context map

Look at the layout one more time, but read each folder as a sentence the AI reads every session:

```text
src/app/
├── core/         ← "These are app-wide singletons. Touch with care."
│   ├── services/
│   ├── interceptors/
│   ├── guards/
│   └── models/
├── shared/       ← "These are reusable, stateless UI primitives."
│   ├── components/
│   ├── directives/
│   └── pipes/
├── features/     ← "Each subfolder is a self-contained business capability."
│   ├── auth/
│   ├── dashboard/
│   └── products/
├── layouts/      ← "App shells and templates. Compose, don't fork."
│   ├── main-layout/
│   └── auth-layout/
├── state/        ← "Global state only. Feature state stays in features/."
└── app.routes.ts
```

That's not just a folder tree. It's a **system prompt written in directories**.

Every time an agent opens this repo, those five top-level folders are doing the work that, in a flat project, I'd be typing into the chat manually every session.

That's the shift that surprised me. I'm not writing a clever prompt. I'm shaping the environment so the prompt becomes obvious.

## What this structure unlocked for me with AI

Once I had `src/app/` carved up this way across my projects, four things became possible that simply weren't before.

**1. Feature-scoped agent runs.** I can scope an agent to `features/checkout/` and trust it won't reach into `features/auth/` or `core/`. The blast radius of a bad refactor is now bounded by a folder. Directory-level IAM scoping for AI.

**2. Per-feature `AGENTS.md` files.** I drop a small markdown file inside `features/checkout/AGENTS.md` describing the domain model, the public surface, and the gotchas. The model reads it automatically when it works in that folder. SAP's open-source `fundamental-ngx` library is already shipping `AGENTS.md` files at root and per-package — this is becoming a convention, not an experiment.

**3. Reproducible scaffolding the model imitates.** Because every feature follows the same shape, my `commands/new-feature.md` (one of the [six files I wrote about previously](/articles/ai-ready-codebase-structure)) tells the model _"scaffold under `features/{name}/`, add a route in `app.routes.ts` lazy-loaded, never touch `core/`."_ The result is structurally identical to every previous feature. Onboarding a new engineer becomes `/new-feature billing`.

**4. A signal-first state convention the model learns from one example.** With Angular 21's `signal()`, `linkedSignal()`, and `resource()`, state often belongs _inside_ the feature folder, not in a global store. The structure makes that visible: a `state/` folder at root for truly cross-cutting state, and feature-local signal stores everywhere else. The model sees the pattern in one feature and replicates it in the next — without me re-explaining my state philosophy in every chat.

## What the flat layout cost me

Before this structure landed across my projects, the flat layout was costing me in ways I only saw in hindsight:

- The model had **no signal of feature ownership** — every change felt global, so suggestions felt global.
- Tribal knowledge ("don't touch the order service, it's haunted") had **nowhere to live** — there was no `features/orders/AGENTS.md` to put it in.
- Refactors **crossed silos** by default — moving a component touched `components/`, `services/`, and `models/` simultaneously, exactly the multi-folder churn that tanks agent quality.
- Every new chat session began with me typing _"in this project, the checkout logic is split between..."_ — context the structure should have been telling the model for free.

The flat layout was fine when I was the only reader. It got expensive the moment the model started reading too.

## The migration path that actually worked for me

I didn't refactor any of my projects in one go. The order that shipped, every time:

1. **Week 1** — Pick one feature. Move it from the flat `components/services/models/` layout into a self-contained `features/{name}/` folder. Update imports. Run tests.
2. **Week 2** — Add an `AGENTS.md` (or `CLAUDE.md`) inside that feature folder. Three paragraphs: what this feature does, what its public surface is, what to never touch.
3. **Week 3** — Add an ESLint or Nx boundary rule that prevents new files in `features/{name}/` from being imported elsewhere except via the feature's barrel.
4. **Week 4** — Repeat for the next feature. The structure emerges from real migrations, not from an upfront redesign.
5. **Ongoing** — Every time I or a teammate or an agent gets confused about where something belongs, that confusion is a missing folder boundary or a missing `AGENTS.md`. I add it.

I'm not refactoring for elegance. I'm refactoring for **readability by two audiences at once** — humans and models — using the same artifacts.

## The new senior frontend engineer

A year ago, the senior Angular engineer was the person who could explain why a feature was lazy-loaded, why `OnPush` mattered, why the auth interceptor sat where it sat. Those skills still matter.

But the senior frontend engineer has quietly become the person who can answer one new question better than anyone else:

> _"If I drop a fresh agent into this repo tomorrow, will it produce code that looks like ours — or code that looks like a generic Angular tutorial from 2019?"_

When the answer is the second one, the folder structure is almost always why.

The layout I've described isn't over-engineering. It's **pre-engineering for the agents I'll be pair-programming with for the next decade**. Every folder boundary I draw is a piece of context I don't have to type again. Every co-located feature is a token I don't have to spend. Every `AGENTS.md` is an onboarding cost I pay once.

My folder structure was always documentation.

In 2026, it's documentation **for two audiences** — and one of them never forgets what it reads.

## If you take one thing away

Open `src/app/` right now.

If you see `components/`, `services/`, `models/` at the root — pick the next feature you'd ship and put it in `features/{name}/` instead. Add an `AGENTS.md` inside it. Commit.

That's it. That's the whole first move. The rest of the structure emerges.
